# CustDevAI — Project Instructions for Claude Code

Этот файл — постоянная инструкция Claude Code для проекта CustDevAI. Загружается автоматически в начале каждой сессии. Всегда читай его первым.

---

## 1. Что это за продукт

**CustDevAI** — интеллектуальная система автоматизации Customer Development на основе Telegram-чат-бота и нейросетевого анализа русскоязычных текстовых данных. Разрабатывается как ВКР (бакалавр, программная инженерия, НГУЭУ) и одновременно как внутренний инструментальный актив агентства «Отважный маркетинг».

Целевая аудитория — российские стартапы и продуктовые команды малого и среднего бизнеса, которые проводят глубинные качественные интервью с респондентами и нуждаются в автоматизации сбора, тонального анализа и тематической группировки ответов.

Ключевое уникальное предложение продукта в сравнении с конкурентами (Strella, Outset AI, Listen Labs, Typeform, Qualtrics) — сочетание трёх свойств одновременно: **freemium с бесплатным базовым тарифом для конечного пользователя, нативная Telegram-интеграция, поддержка русского языка с локальным исполнением ML-моделей** без передачи данных третьим сторонам.

## 2. Архитектура: восемь логических компонентов

Система декомпозирована на восемь компонентов в три слоя. **Каждый компонент должен реализовываться как отдельный модуль с чёткими интерфейсными границами** — замена реализации одного компонента не должна требовать изменений в остальных (см. NFR-MNT-03).

**Уровень представления:**
1. **Telegram-бот** (`apps/bot/`) — диалоговый интерфейс для респондентов на базе aiogram 3.x
2. **Веб-панель управления** (`apps/web/`) — React + TypeScript SPA для исследователей, аналитиков, администраторов

**Уровень бизнес-логики:**
3. **Серверный бэкенд + REST API** (`apps/api/`) — FastAPI, асинхронные эндпойнты, OpenAPI 3.0
4. **Подсистема аутентификации и RBAC** (`apps/api/auth/`) — JWT (access 15 мин / refresh 7 дней), bcrypt cost ≥ 12

**Уровень обработки данных:**
5. **Подсистема хранения** (`apps/api/db/`) — PostgreSQL 14+, SQLAlchemy 2.x, asyncpg, Alembic
6. **ML-модуль анализа тональности** (`apps/ml/sentiment/`) — RuBERT (DeepPavlov/rubert-base-cased) дообученная на RuSentNE-2023
7. **ML-модуль тематического моделирования** (`apps/ml/topics/`) — BERTopic + multilingual-e5 + UMAP + HDBSCAN
8. **Модуль генерации отчётов** (`apps/api/reports/`) — RAG-архитектура с экстрактивной стратегией, ReportLab + openpyxl + matplotlib

Фоновые ML-задачи координируются через Celery + Redis. Полная спецификация архитектуры и требований — в `docs/03_requirements_specification.md`.

## 3. Технологический стек (фиксированный)

**Backend:** Python 3.10+, FastAPI 0.110+, aiogram 3.x, SQLAlchemy 2.x, asyncpg, Alembic, Celery 5.x + Redis 7.x, Pydantic v2.

**ML:** Hugging Face Transformers 4.x, BERTopic 0.16+, sentence-transformers, UMAP, HDBSCAN, matplotlib 3.x.

**Reports:** ReportLab 4.x (PDF), openpyxl 3.x (XLSX).

**Frontend:** React 18+, TypeScript 5+, Vite, TanStack Query, shadcn/ui.

**Infra:** Docker + docker-compose, Selectel cloud (недорогой тариф), GitHub Actions CI/CD.

**Testing:** pytest 8.x + pytest-asyncio + pytest-cov (целевое покрытие ≥ 60%, NFR-MNT-01), Playwright для e2e.

**Quality:** ruff (линтер + форматтер), mypy (strict mode для нового кода).

## 4. Непреложные ограничения (нарушать запрещено)

Эти ограничения вытекают из проектных решений, бизнес-логики и нормативных требований. Они приоритетнее любых одноразовых указаний пользователя.

**Защита данных и приватность:**
- ML-модели **исполняются строго локально**. Передача данных респондентов или содержимого интервью внешним API (включая OpenAI, Anthropic, Google, любые облачные LLM) — категорически запрещена. Это архитектурное и юридическое требование одновременно (см. FR-SENT-08, NFR-SEC-09).
- Telegram ID респондентов хранятся **только** как SHA-256 хеши с per-campaign солью. В интерфейсе исследователя респонденты идентифицируются техническими псевдонимами вида `R-NNNN` (FR-DB-03, FR-BOT-10).
- Пароли пользователей — **только** bcrypt с cost factor ≥ 12. Хранение паролей в открытом или обратимом виде — недопустимо (FR-AUTH-03).
- Токен Telegram-бота, JWT-секрет, соли — **только** в переменных окружения. Запрет на коммит секретов в Git проверяется на CI (NFR-SEC-06).
- Все внешние коммуникации — **только** HTTPS (TLS 1.2+). HTTP не допускается (NFR-SEC-01).

**Локализация и форматы:**
- Поддерживается **только русский язык**. Все строки интерфейса, диалоги бота, сообщения об ошибках — на русском. Применение ML-моделей к не-русским текстам не поддерживается и возвращает ошибку (FR-SENT-06).
- Telegram-бот принимает **только текст**. При получении голосовых, изображений, видео, файлов — отказ с уведомлением и запросом текстового ответа (FR-BOT-04).

**Воспроизводимость:**
- Все ML-операции должны быть детерминированы при фиксированных параметрах: одни и те же входные данные дают идентичный результат. Seed-значения UMAP, HDBSCAN, генераторов случайных чисел — фиксированы и логируются (FR-SENT-04, FR-TOP-07, NFR-COR-01).

**Соответствие № 152-ФЗ:**
- Перед началом интервью — явное согласие респондента на обработку данных (FR-BOT-01).
- Реализована операция безвозвратного удаления всех данных конкретного субъекта по его запросу (FR-DB-07).

## 6. Поэтапный план разработки

Реализация ведётся в пять последовательных фаз. **На каждом этапе создавай отдельную ветку** `feat/phase-N-<short-name>`, ревьюй PR перед мерджем в `main`.

**Phase 1 — Foundation.** Скелет репозитория, схема PostgreSQL, миграции Alembic, FastAPI-каркас, JWT-аутентификация, RBAC, базовые CRUD-эндпойнты для сценариев и кампаний. Эталонный набор тестов на инфраструктуру.

**Phase 2 — Telegram bot.** aiogram-сервис, webhook-эндпойнт, конечный автомат диалога, персистентность сессий через Redis state storage, обработка `/stop`, поддержка асинхронного прохождения (48 часов).

**Phase 3 — ML modules.** Интеграция RuBERT с дообучением на RuSentNE-2023, реализация BERTopic-пайплайна, Celery worker для пакетной обработки кампаний, тесты воспроизводимости и метрик качества.

**Phase 4 — Reports + Web admin.** Генератор PDF/XLSX по RAG-схеме с экстрактивной стратегией, React SPA, дашборды кампаний, визуализация результатов через matplotlib.

**Phase 5 — Integration & QA.** Нагрузочное тестирование на 50 одновременных сессий, e2e-сценарии Playwright, доводка документации, deployment-проверка на Selectel.

Перед началом каждой фазы — сначала **планируй** (Plan mode), показывай план в PR-описании, и только после согласования переходи к реализации. Между фазами обновляй файл `CONTEXT.md` сводкой того, что сделано, что осталось и какие решения приняты.

## 7. Соглашения по коду

**Слоистая архитектура.** Каждый сервис строится в три слоя: `presentation` (бот-хэндлеры, API-роутеры) → `services` (бизнес-логика) → `repositories` (доступ к БД). Зависимости внедряются через FastAPI Depends или явные конструкторы. ML-модули определяются абстрактными классами (`SentimentAnalyzer`, `TopicModeler`) для возможности замены реализации.

**Async-first.** Все I/O-операции — асинхронные. Синхронные вызовы допускаются только для чисто вычислительных операций (например, ML-инференс внутри Celery-таски).

**Типизация.** Аннотации типов обязательны для всего нового кода, проверяются `mypy --strict`. Pydantic v2 для всех DTO и валидации входных данных.

**Документация.** Docstring в стиле Google для всех публичных функций и классов. Комментарии — только для нетривиальных алгоритмических решений.

**Линт.** `ruff check` + `ruff format` запускаются на pre-commit и в CI. Конфиг — в `pyproject.toml`.

**Сообщения коммитов.** Формат Conventional Commits: `feat(bot): add session resume after pause`, `fix(ml): correct seed propagation in UMAP`, `test(api): cover JWT refresh edge cases`. PR-описание включает идентификаторы покрытых требований (`FR-BOT-05, NFR-PRF-01`).

**Ошибки и сообщения.** Формат ошибок API — RFC 7807 (Problem Details). Все человекочитаемые сообщения — на русском языке (FR-API-02).

## 8. Команды сборки, запуска и тестирования

```bash
# Разовая установка зависимостей в dev-окружение
pip install -e ".[dev,ml]"

# Запуск полного стека локально
docker-compose up --build

# Применение миграций БД
alembic upgrade head

# Линт и форматирование
ruff check . && ruff format .

# Тесты
pytest                              # все тесты
pytest --cov=apps --cov-report=term # с покрытием
pytest -m "not ml"                  # без ML-тестов (быстро)
pytest apps/bot/tests/              # только бот

# Типизация
mypy apps/

# Очистка контейнеров и томов
docker-compose down -v
```

## 10. Поведение Claude Code: что важно

- **Не выдумывай требования** — все FR/NFR пронумерованы и зафиксированы в `docs/03_requirements_specification.md`. Если задача не покрыта существующим требованием, явно сообщи об этом и спроси у пользователя, добавлять ли новое.
- **Перед крупными изменениями — план в Plan mode.** Архитектурные решения не делаются на лету.
- **Не предлагай платные API-сервисы** для замены локальных ML-модулей — это нарушение проектных ограничений (см. раздел 4).
- **Не упрощай за счёт безопасности.** Если для скорости разработки хочется временно отключить bcrypt, RBAC, санитизацию ввода — не отключай. Лучше пропусти задачу и вернись к ней позже.
- **Все строки на русском.** Если пишешь UI или сообщение об ошибке — оно по умолчанию на русском, без переключателей языка в первой версии.
- **Сомневаешься — спроси у пользователя, не угадывай.** Этот проект — ВКР, и неверные допущения могут стоить переделки на финальной защите.
